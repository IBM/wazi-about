# Adding more file associations

IBM Z Open Editor registers itself in VS Code with support for several languages. For each language it also defines a set of file extensions that by default will be mapped to the registered languages as shown in the table below.

Language VS Code Language ID Default File Extensions
Enterprise COBOL cobol .cbl .cpy .cob .copy .copybook .cobol .cobcopy
Enterrise PL/I pl1 .pli .pl1 .pc .pci .plinc .pcx .inc .include
Z High Level Assembler hlasm .s .assemble .asm .hla .hlasm
JCL jcl .jcl .cntl .proc

When you open a file with one of these extensions in Z Open Editor then VS Code will display the mapped language in the language selector on the bottom-right of the status bar. You can click this bar to temporarily switch the open editor to any other language available in VS Code.

If you want to use a different file extension in addition to these for any of these languages then you can define them yourself in the user or workspace settings.

  • Open the Preferences editor and switch to either user or workspace settings. Workspace settings would be useful to define the mappings locally to the current project and store it in SCM with your programs.
  • In the Search bar at the top type associations.
  • This will bring up the VS Code "Files: Associations" setting that contains a list of the current non-default file associations defined. The default associations shown in the table above are not listed here as they cannot be changed.
  • Click the Add Item button to add another entry. It will prompt for a "Key", which is the file extension starting with the symbols *. such as *.cp1 and a "Value" entry that is the language identifier as listed in the table above.

As an alternative to using the VS Code Preferences UI you can also edit the JSON of the user or workspace settings file. The format would be

"files.associations": {
    "*.cp1": "cobol"
}

The format of the key is very flexible as you can enter a full GLOB pattern here allowing you to map files to a language by not only using file extensions, but arbitrary substring mappings as well. This ability will be used for remote file mappings as described further below.

Note, if you remove some of the entries there that were added by Z Open Editor at startup such as "*.COBOL*": "cobol", and save, they might reappear afterwards. To prevent that see the details below on managing remote file associations that cause this behavior.

# Defining Include file associations

On the page Setting property groups we described how you can define so-called Property Groups that tell Z Open Editor where to look for include files such as COBOL Copybooks. These include files can be located on the local file system or could be in remote MVS data sets. If you store them locally then another set of default file extensions exist for identifying a valid include file for each language. You can find the default sets of include file extensions in the Settings as well and use these to customize your own file extensions to be used. Without a valid file extension Z Open Editor will not be able to find include files and not open them correctly for editing as well.

  • Open the Preferences editor and switch to either user or workspace settings. Workspace settings would be useful to define the mapping locally to the current project and store it in SCM with your programs.
  • On the left of Settings editor expand the "Extensions" group and select the entry "IBM Z Open Editor".
  • Scroll on the right to see all the different Z Open Editor settings available. There are three settings available for configuring include file extensions:
    • Cobol: Include File Extensions
    • Hlasm: Include File Extensions
    • Pl1: Include File Extensions
  • Each shows a list of the currently registered file extensions populated with a set of defaults.
  • You can now add new extensions to the list.
  • Click the Add Item button and you can enter the extension with a leading dot-character such as .cp1.
  • Once you entered a new extension and save you will see that your new extension has also been added automatically to the files.associations setting that was discussed above.
  • If you remove an extension from the list then Z Open Editor will not currently remove it form the files.associations however. You have to remove it there yourself.

Once configured, Z Open Editor will now find include files with these new extensions in the locations specified with your Property Groups. When you open these files, they will also be correctly opened in the editor as include files for the assigned language. This also means for COBOL and PL/I the files will be correctly parsed as include files and not full program files. See here for more details on the specific differences.

# Remote file associations

IBM Z Open Editor integrates with and provides advanced capabilities for Zowe Explorer to edit program files from remote z/OS locations. You can,for example, find your COBOL programs in Zowe Explorer's MVS explorer and the open them directly from there for editing in Z Open Editor. As MVS data sets do not have file extensions, which are essential for VS Code to identify the language of file opened in the editor, Z Open Editor gives you special settings and capabilities to configure the editor for MVS.

Within the family of zopeneditor.datasets VS Code settings defined by Z Open Editor, you can specify patterns for MVS data set names that should contain COBOL, PL/I, or HLASM programs, copybooks, includes, and JCL. This means that when you open members of these data sets using Zowe Explorer in the editor, they shall be considered the designated type of file. This is accomplished in Z Open Editor by allowing you to specify name patterns that are then mapped to VS Code files.associations settings as described above for local files as well. To edit these name patterns or add to them you use the VS Code Preferences editor.

  • Open the Preferences editor and switch to either user or workspace settings. Workspace settings would be useful to define the mapping locally to the current project and store it in SCM with your programs.
  • On the left of Settings editor expand the "Extensions" group and select the entry "IBM Z Open Editor".
  • Scroll on the right to see all the different Z Open Editor settings available. There are four settings available for configuring data set names:
    • Datasets: Cobol Datasets
    • Datasets: Hlasm Datasets
    • Datasets: Jcl Datasets
    • Datasets: Pl1 Datasets
  • Each shows a list of the currently registered names populated with a set of defaults.
  • You can now add new name pattern to the list by clicking the Add Item button. As you see in the list of defaults, it is a good idea to include dots to make the mappings to MVS data set names less ambiguous.
  • Once you entered a new entry and save you will see that your new name has also been added with an additional * wildcard at the end automatically to the files.associations setting that was discussed above.
  • If you remove a entry from the list then Z Open Editor will not currently remove it form the files.associations however. You have to remove it there yourself afterwards.

Also note, that before you be able to remove any of the mappings from the files.associations you need to remove them first from the datasets as well as includeFileExtensions settings first as Z Open Editor tries to automatically add its settings into the associations to enable its features.

You can also edit all these settings directly in the JSON settings files.

The result of these settings now is, for example, when you open any member of IBMUSER.SAMPLE.COBOL in MVS, the content of the member is opened in a COBOL editor, and likewise for PL/I, HLASM, and JCL. When opening files the editor will follow the following rules.

Rule 1: By default, members in the COBOL data set that end with .COBOL, .COBCOPY, or .COPYBOOK are supposed to be COBOL, COPYBOOKs, or COBCOPY. Members in the PL/I data set that end with .PLI, .PLINC, or .INCLUDE are supposed to be PL/I, PLINC, or INCLUDES. Members in the HLASM data set that end with .ASM are supposed to be HLASM. If your team or company uses COPYBOOK data sets for HLASM copymembers, you can move those settings from cobolDatasets to the hlasmDatasets parameter.

Rule 2: The data sets are listed in a mapping files.associations in the preferences.

"zopeneditor.datasets.cobolDatasets": [
    ".COBOL",
    ".COB",
    ".COBCOPY",
    ".COPYBOOK",
    ".COPY"
],
"zopeneditor.datasets.jclDatasets": [
    ".JCL"
],
"zopeneditor.datasets.pl1Datasets": [
    ".PL1",
    ".PLI",
    ".INC",
    ".INCLUDE"
],
"zopeneditor.datasets.hlasmDatasets": [".ASM"]

Alternatively, you can directly specify an array of associations instead of using the "zopeneditor.dataset" property above. For example,

"files.associations": {
  "*.COBOL*": "cobol",
  "*.COB*": "cobol",
  "*.COBCOPY*": "cobol",
  "*.COPYBOOK*": "cobol",
  "*.COPY*": "cobol",
  "*.PL1*": "pl1",
  "*.PLI*": "pl1",
  "*.INC*": "pl1",
  "*.INCLUDE*": "pl1",
  "*.JCL*": "jcl",
  "*.ASM*": "hlasm"
}

Again, if you prefer COPYBOOK data sets be opened as hlasm data sets, you can designate that in this setting.

Such a setting is used to recognize any file name, including data set and member names, which contain the noted strings as the appropriate type of file.

Using this setting, you can navigate to the path specified, and see a downloaded copy of all remote files edited via the Zowe VS Code extension.

Note: All file temps are deleted on startup and shutdown.
